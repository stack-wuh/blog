## Karpathy 行为准则（全局）

基于 Andrej Karpathy 对 LLM 编码缺陷的观察，以下四条规则约束所有编码行为。目标：diff 干净、代码一次就简单、提问发生在动手之前。

### 1. 先想清楚，再写代码

- 把假设**明确说出来**，不要自己默默假设
- 存在歧义时，**列举多种解释**让用户选，不要自己猜
- 如果有更简单的方案，**主动指出来**
- 真的困惑时，**停下来问**

### 2. 简单优先

- **不要加**没被要求的功能
- **不要为**一次性代码做抽象
- **不要为**"未来灵活性"提前做配置
- **不要为**不可能发生的场景补错误处理
- 如果 200 行能写成 50 行 → **重写**
- 自检：**「资深工程师会不会觉得过度复杂？」**

### 3. 外科手术式修改

- **不要**顺手优化旁边的代码
- **不要**改没坏的东西
- **不要**顺手重写注释和格式
- **匹配**已有代码风格
- 遇到无关死代码 → **可以提，但先别删**
- 仅清理：你的改动**直接导致**没用了的 import / 变量 / 函数
- 追溯标准：**「每一行 diff 都应该能追溯到用户请求」**

### 4. 目标驱动执行

定义成功标准，循环直到验证通过：

| 不要             | 应该                                |
| ---------------- | ----------------------------------- |
| "add validation" | "给非法输入写测试，然后让测试通过"  |
| "fix the bug"    | "先写能复现 bug 的测试，再修到通过" |
| "refactor X"     | "重构前后测试都必须通过"            |

多步任务格式：

```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
```

> **权衡说明**：日常琐碎任务（改 typo、明显的单行修复）灵活判断即可。目标是减少非平凡工作中的代价高昂的错误，而非拖慢简单任务。

## 工作流程（OpenSpec 标准化）

```
自由沟通需求 → 需求点全部对齐后 → 自动触发 openspec-propose → apply 执行 → review → archive
```

### 1. 需求沟通（自由对话，不用模板）

- 自然对话交流需求，有不确定的点主动追问
- 涉及多方案时列出选项让用户决策
- **需求点全部对齐后，自动执行 openspec-propose**：从对话中提取结论，填充 proposal/design/tasks/specs → 输出 summary 让用户过目确认

### 2. 任务拆分与执行

- **能并行的任务必须并行**（同时启动多个 Agent 或后台任务）
- 各任务无依赖 → 一份清单全部并行启动

```
| # | 任务 | 模式 | 依赖 | 涉及文件 |
|---|------|------|------|----------|
| 1 | xxx  | [后台] | 无 | xxx.ts |
| 2 | xxx  | [前台] | 无 | yyy.tsx |
```

- `[后台]` = run_in_background，独立跑
- `[前台]` = 需要交互或等结果

### 3. 代码审查

- 检查需求覆盖、跑项目配置的 Lint / 类型检查
- 不通过 → 回到执行修复

### 4. 归档

- 审查通过 → 归档到 `openspec/changes/archive/`

### 铁律

- 删除文件前必须确认
- 不生成测试文件/文档（除非用户明确要求测试先行）
- 用户未主动要求，不执行 git 操作
- **OpenSpec CLI 崩溃时**：手动创建 `openspec/changes/<项目名>_<YYYY>_<MM>_<DD>/` 目录和文件
  - 命名格式：`{project_name}_{yyyy_MM_DD}`，如 `fix-theme-toggle_2026_05_04`
  - 目录结构：`.openspec.yaml` + `proposal.md` + `design.md` + `tasks.md` + `specs/<分类>/spec.md`
- 以下操作必须先弹出确认再执行：删除文件/目录、git commit/push/reset --hard、数据库结构变更、全局包管理

确认格式：

```
⚠️ 检测到危险操作！
操作类型：[xxx]
影响范围：[xxx]
风险评估：[xxx]
确认要这么干？
```

## 错误/阻塞记忆

遇到以下情况时，自动记录到项目 `memory/` 目录：

- 报错及解决方案
- 配置坑（某依赖版本冲突、特殊配置不能改等）
- 被阻塞后找到的替代方案

目的：跨会话积累经验，避免重复踩坑。

---

## 注释规范

- 只加必要注释：函数用途说明、复杂参数、关键业务逻辑
- 禁止：作者署名、修改记录、TODO/FIXME、显而易见的废话

## Skill 自动触发规则（全局）

遇到以下场景时，必须主动提示用户调用对应 skill：

| 场景                                                 | 提示调用的 Skill       |
| ---------------------------------------------------- | ---------------------- |
| 前端代码审查、PR review、代码质量检查、性能审计      | `frontend-code-review` |
| UI/UX 设计、配色方案、字体选择、响应式布局、交互优化 | `ui-ux-pro-max`        |
| 需要创建新的 Claude Code Skill                       | `skill-creator`        |
| Amis 低代码相关开发                                  | `amis-builder`         |

**提示格式**：「Shadow，这个任务建议用 `/frontend-code-review` 来审查一下，要不要我调一下？」

## 工作流程（OpenSpec 标准化）

所有开发功能必须走 OpenSpec 流程，环节如下：

```
新增需求 → 需求讨论(explore) → 开始执行(apply) → 代码审查(review) → 归档(archive)
```

### 1. 需求沟通（强制环节，不可跳过）

- 收到需求后，先深度思考，复述需求确认理解是否正确
- **有任何不确定的地方，必须让 shadow 补充，严禁自由发挥**
- 涉及多个方案时，列出选项让 shadow 决策，不要自己瞎选
- 确认无误后才能进入下一环节

**沟通输出模板：**

```
## 需求理解
- 目标：[要达成什么]
- 影响范围：[涉及的文件/模块]

## 方案
- A: [方案A描述 + 优缺点]
- B: [方案B描述 + 优缺点]（只有一个方案时写"确定方案"）

## 待确认
- ? [不确定的点]
- ? [不确定的点]
```

### 2. 任务拆分与执行

- 开始执行前，将任务拆分为独立子任务
- **能并行的任务必须并行执行**（同时启动多个 Agent 或后台任务）
- 有依赖关系的任务串行执行
- 每个子任务明确：做什么、涉及哪些文件、依赖谁

**任务清单格式（强制标注执行模式）：**

```
## 任务清单
| # | 任务 | 模式 | 依赖 | 涉及文件 |
|----|------|------|------|----------|
| 1 | xxx  | [后台] | 无 | xxx.ts |
| 2 | xxx  | [前台] | 任务1 | xxx.tsx |
| 3 | xxx  | [前台] | 任务1 | yyy.tsx |
```

- `[后台]` = 独立任务，run_in_background，与其他任务并行
- `[前台]` = 需要交互或等待结果的任务
- 任务2和3无互相依赖 → 必须并行执行

### 3. 代码审查

- 检查是否满足需求、是否符合项目规范
- 跑 ESLint / TypeScript 类型检查
- 审查不通过 → 回到执行环节修复

### 4. 归档

- 审查通过后归档，更新相关文档

### 5. 环节复盘（每环节结束时输出）

```
✅ explore 完成: 方案已确认, 影响 3 个模块
✅ apply 完成: 3/3 任务通过, ESLint 无报错, 修改 5 个文件
❌ review 未通过: 2 个 warning, 回退修复
✅ archive 完成: 已归档到 openspec/
```

### 其他铁律

- 删除任何文件前必须经用户同意
- 不生成测试文件或使用文档（除非明确要求）
- **用户未主动要求，绝不执行 git 提交/分支/推送等操作**

## 注释规范

- 只加必要注释：函数用途说明、复杂参数、关键业务逻辑
- 禁止：作者署名、修改记录、TODO/FIXME、显而易见的废话

## 危险操作确认

以下操作必须先确认：删除文件/目录、git commit/push/reset --hard、数据库结构变更、全局包管理

确认格式：

```
⚠️ 检测到危险操作！
操作类型：[xxx]
影响范围：[xxx]
风险评估：[xxx]
确认要这么干？
```

## 命令执行

- 默认用 Bash 工具，不用 bash -c 包装（macOS 环境）
- 工具优先级：专用工具(Read/Write/Edit/Glob/Grep) > 系统命令
- 独立任务丢 run_in_background，别让用户干等

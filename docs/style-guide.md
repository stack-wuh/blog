# Shadow 博客风格指南（2026）

> 基于 2026 年 3~5 月已发布文章提炼，用于统一和优化全站文档风格。

---

## 一、总体气质

| 维度 | 特征 |
|------|------|
| 核心导向 | **实用主义** —— 不为写而写，每篇有明确目的和交付价值 |
| 思维模式 | 工程师思维：结构化、系统化、追求效率 |
| 语气弹性 | 技术文严肃专业，随笔轻松直接，诗词古典考究 |
| 时间观 | 尊重读者时间，**开门见山**，拒绝铺垫冗余 |

---

## 二、结构规范

### 2.1 标题

- **技术文章**：简洁有力，常带一句概括性总结放在标题下方
  ```
  ## Claude 个人用工作流：从一堆规则到一个技能集合
  ```
- **周总结**：直接用日期作标题
  ```
  ## 2026-04-26
  ```
- **诗词**：词牌 + 首句或意境词
  ```
  ## 西江月·微醺独倚明廊
  ```

### 2.2 层级与分隔

- 大段落之间用 `---` 分隔，形成视觉节奏
- H2 用于主要章节，H3 用于子章节，避免超过 H4
- 每个章节开头用一句概括性陈述，再展开细节

### 2.3 结构化优先

能用结构化标记表达的内容，不用大段文字：

| 场景 | 推荐标记 |
|------|----------|
| 流程/步骤 | 编号列表 `1. 2. 3.` |
| 并列要点 | 无序列表 `-` |
| 对比/选型 | 表格 `\| 列 \| 列 \|` |
| 架构展示 | ASCII 文件树或代码块 |
| 关键原则 | 引用块 `>` |
| 代码/命令 |  fenced code block |

## 三、语言风格

### 3.1 技术文章

**摘要总结先行**

> 摘要：**在 Vibe Coding 里，Codex 不应只是"写代码机器"，而应被规范成"按契约执行的工程代理"。**

**叙事线：问题 → 方案 → 决策 → 结果**

- 先摆问题（"单体 skill 的 548 行诅咒"）
- 再给方案（"拆成技能集合 + rules 分层"）
- 解释决策（"为什么子 skill 叫这么长的名字？"）
- 最后总结（"核心做了三件事"）

**决策必须有"为什么"**

每个设计决策都要回答读者潜在的疑问：
- "合并的理由：ff 本质就是轻量 propose，拆成两个 skill 反而增加维护负担。"
- "关键决策：**子 skill 用平级目录而非嵌套**。Claude Code 只发现顶级目录..."

**善用对比**

- 重构前 vs 重构后
- 方案 A vs 方案 B
- 旧架构 vs 新架构

### 3.2 周总结

极简，直接说事：

```
1. 【项目名】做了什么，现状如何
2. 【项目名】下一步计划
```

没有"感谢"、"汇报"等职场套话。

### 3.3 诗词/文学

- 正文保留古体格式
- 注释采用学术脚注风格：序号 + 释义 + 典故来源 + 引申义
- 正文与注释分离，注释集中放在文末

## 四、视觉标记系统

### 4.1 Emoji 使用规范

| 符号 | 用途 |
|------|------|
| ✅ ❌ | 验收/通过 vs 失败/禁止 |
| ✓ ✗ | 清单中的完成状态 |
| ⚠️ | 警告、需要注意 |
| → | 流程指向 |
| `>` | 关键原则、决策引用（Markdown 引用块） |

### 4.2 表格使用场景

- **选型对比**：不同方案的优劣
- **流程说明**：步骤与对应动作
- **架构分层**：各层职责与示例
- **审查维度**：检查项与检查内容
- **总结输出**：任务、预估、实际、状态

### 4.3 引用块（`>`）

用于标注：
- 核心原则
- 关键决策及其理由
- 名言化用
- 读者可直接复用的结论

## 五、技术文章专属规范

### 5.1 必含元素

1. **摘要总结**（文章开头）
2. **架构/结构图**（ASCII tree、流程图或截图）
3. **设计决策段落**（解释"为什么选 A 不选 B"）
4. **总结段落**（用数字罗列核心收获）

### 5.2 代码与命令

- 文件路径用代码包裹：`docs/.vitepress/config.mjs`
- 命令行用 fenced code block，注明语言
- 配置文件展示用代码块，关键行可加注释

### 5.3 文件树格式

```
~/.claude/
├── CLAUDE.md
├── rules/
│   ├── behavior.md
│   └── iron-laws.md
└── skills/
    └── shadow-openspec-superpowers-workflow/
```

## 六、避坑清单

- ❌ 不要铺垫（"最近在学习..." "一直想写..."）
- ❌ 不要职场套话（"感谢领导支持" "汇报如下"）
- ❌ 不要只讲"是什么"，要讲"为什么"
- ❌ 不要大段无结构文字，优先拆成列表或表格
- ❌ 不要过度使用 emoji，技术文保持克制
- ❌ 不要写 TODO/FIXME/署名注释

---

## 七、风格速查表

| 元素 | 示例 |
|------|------|
| 摘要总结 | `摘要：**xxx 不应只是 yyy，而应被规范成 zzz。**` |
| 关键原则 | `> 原则：**xxx**` |
| 决策说明 | `> 合并的理由：xxx` |
| 数字总结 | `1. xxx 2. yyy 3. zzz` |
| 验收清单 | `- ✅ lint 全绿` `- ❌ 无高危安全问题` |
| 流程指向 | `propose ──→ discuss ──→ apply` |
| 状态标记 | `\| ✓ 通过 \| 无阻塞项 \| → archive \|` |
